<objective>
Capture completed Task subagent reports inside `runAgent`'s for-await SDK loop so they survive both the abort path and the success path via a new `subagentResults` field on `AgentResult`. Today these arrive on `user` events as `tool_result` blocks but the loop only inspects `system`, `assistant`, `stream_event`, and `result` events — the payloads are dropped on the floor.

Purpose: When `AGENT_TIMEOUT_MS` fires mid-flight, downstream code in `bot.ts` needs the actual subagent payloads (not just the `task_notification.summary` already streamed to Telegram) so the timeout reply can show real research output instead of a static "task was too complex" message.

Output: A new closure-scoped buffer in `runAgent`, a new `user`-event handler in the for-await loop, and an extended `AgentResult` interface — surfaced on both return paths.
</objective>

Do NOT touch `runAgentWithRetry`, `recoverFromCompactionIfNeeded`, the typing interval, or any other unrelated code. Do NOT change `AGENT_TIMEOUT_MS` or any abort-related plumbing.

Edit 1 — Add new exported type and extend AgentResult (around src/agent.ts:157-162):

Add a new exported interface `SubagentResult` immediately above the existing `AgentResult` interface. Shape per D-decision "Tool_result content shape":
- `toolName: string` — the name of the tool whose result this is (e.g. `'Task'`)
- `toolUseId: string` — the tool_use_id that paired the assistant tool_use with this user tool_result
- `content: string` — extracted text. If `tool_result.content` is a string, use as-is. If it is an array of `{ type: 'text', text: string }`, join the `text` fields with `\n`. If neither shape, fall back to `JSON.stringify(rawContent)`.
- `isError: boolean` — mirrors `tool_result.is_error ?? false`

Then extend `AgentResult` with one new optional field: `subagentResults?: SubagentResult[]`. Optional means no breaking change for any other caller.

Edit 2 — Add closure variables in runAgent (around src/agent.ts:227-235, immediately after the existing `let streamedText = '';` line):

Add two new closure variables:
- `const toolUseRegistry = new Map<string, { name: string; input: unknown }>()` — keyed by `tool_use_id` from assistant tool_use blocks
- `const subagentResults: SubagentResult[] = []` — accumulates SubagentResult objects across user tool_result events for the lifetime of this turn

These are `const` because the contents mutate but the references do not.

Edit 3 — Populate toolUseRegistry in the existing assistant-event branch (extending the iteration at src/agent.ts:324-333):

Inside the existing `if (ev['type'] === 'assistant')` block, where `content` is already walked for `tool_use` blocks to emit `onProgress`, also call `toolUseRegistry.set(block.id, { name: block.name, input: block.input })` for every tool_use block that has both `id` and `name`. Tighten the inline cast on `content` so TypeScript sees `id` and `input` as well — keep it loose (cast to `Array<{ type: string; name?: string; id?: string; input?: unknown }>`). Do not break the existing `onProgress` call.

Edit 4 — Add a new user-event handler immediately after the stream_event block and before the result block (insert between current src/agent.ts:365 and :367):

Handle `ev['type'] === 'user'` by walking `ev['message']?.['content']` (which is an `Array<Record<string, unknown>>` of mixed blocks). For each block where `block.type === 'tool_result'`:
1. Read `tool_use_id` (string), `is_error` (boolean | undefined), and `content` (string | Array<{type:'text', text:string}> | unknown).
2. Look up `toolUseRegistry.get(tool_use_id)`. If not found, skip (we never registered it — likely a non-Task tool or out-of-order event).
3. **Gate on `registered.name === 'Task'` per CONTEXT decision** — Task-only capture for this phase. (Per CONTEXT "Where to capture tool_result blocks" + the discretion note in the debug map under "Implementation surface area / NEW after line 365".)
4. Normalize content to a string per the shape rules in `SubagentResult.content` above (string passthrough; array-of-text-blocks joined by `\n`; otherwise `JSON.stringify`).
5. Push a fully populated SubagentResult onto `subagentResults`. Trim the final content string before pushing only if necessary — DO NOT truncate or sanitize. Truncation/formatting is the bot.ts responsibility (Plan 07-02).

Wrap the per-block extraction in a tiny `try { ... } catch { /* skip malformed block */ }` so a single weird payload cannot kill the loop. Match the project's "best-effort" file/event handling posture (see scratchpad.ts:52-55 and media.ts:182-194 for the convention).

Edit 5 — Surface on both return paths:
- Abort return at src/agent.ts:417 — change from `return { text: null, newSessionId, usage, aborted: true };` to `return { text: null, newSessionId, usage, aborted: true, subagentResults };`.
- Success return at src/agent.ts:432 — change from `return { text: resultText, newSessionId, usage };` to `return { text: resultText, newSessionId, usage, subagentResults };`.

If `subagentResults.length === 0` we still return the empty array (not `undefined`). That makes downstream code at bot.ts simpler — it can call `.length` without optional-chaining. The field on AgentResult stays declared `?:` for backward compatibility with any existing test or external consumer.

Do NOT log subagent payloads to the structured logger (privacy — these can contain sensitive research). One terse INFO log at the end of the abort branch is fine: `logger.info({ subagentResultsCount: subagentResults.length }, 'Agent query aborted by user')` — modify the existing log line at src/agent.ts:416 in place, do not add a new one.
